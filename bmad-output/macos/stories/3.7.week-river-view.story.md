## Dev Notes
<!-- LOCKED. External dev tools must not edit Dev Notes. -->

- **Câu SQL đề xuất** [Inference — chưa chạy thử, dev agent verify lại bằng `sqlite3` CLI hoặc test tay trước khi khoá]:
  ```sql
  SELECT date(ts, 'unixepoch', 'localtime') AS day,
         AVG(send_risk) AS avg_risk,
         COUNT(*) AS cnt
  FROM mood_events
  WHERE event_type = 'sample' AND ts >= ? AND ts < ?
  GROUP BY day
  ORDER BY day ASC;
  ```
  Bind `?` = epoch giây của "00:00 6 ngày trước" và "00:00 ngày mai" (giống pattern `startOfDay`/`startOfTomorrow` đã dùng ở `FetchTodaySamples`, dòng 407-410). SQLite's `date(ts, 'unixepoch', 'localtime')` cần xác nhận build có bật đúng timezone local — nếu lệch, đối chiếu cách `FetchTodaySamples` đang tính `startOfDay` bằng `NSCalendar currentCalendar` (Objective-C side) thay vì SQL side, để tránh 2 nơi tính "ngày" khác nhau.

- **Lấp khoảng trống ngày-không-có-mẫu — làm ở TẦNG HÀM FETCH, không đẩy xuống caller** [Inference, cải tiến so với pattern cũ]. `FetchTodaySamples` hiện để CALLER tự phát hiện gap bằng so sánh `ts` liên tiếp (lặp lại logic này ở CẢ `ReflectionScreenMac.mm` dòng 118-125 LẪN `SettingsWindowController.mm` dòng 447-453 — 2 nơi cùng 1 logic). Với hàm MỚI `FetchWeekSamples`, đề xuất hàm TỰ duyệt đủ 7 ngày (kể cả ngày SQL không trả về hàng nào) và tự chèn `NSNull` cho ngày thiếu, trả về LUÔN đủ 7 phần tử theo thứ tự — caller không cần tự phát hiện gap nữa. Giảm 1 lớp trùng lặp logic thay vì lặp lại lần 3.

## Testing
<!-- LOCKED. External dev tools must not edit Testing. -->

- Cổng tự động: `make test` xanh, build sạch, `make brand-lint` 0.
- **Mở app thật bấm-thử bắt buộc** (luật Epic 3).
- **Key scenarios:** đủ 7 ngày có dữ liệu → sông tuần vẽ 7 điểm liền mạch; có 1-2 ngày không dùng app → gap hiện đúng vị trí (không nội suy); chuyển qua lại Ngày↔Tuần nhiều lần → không leak state/không giữ dữ liệu cũ.
- **Per-AC verification:** AC1 → query trực tiếp bằng `sqlite3` CLI (sau khi giải mã file test — xem cách `MoodStoreMac.mm` OpenWorkingDB dùng file tạm plaintext) đối chiếu kết quả hàm trả về; AC2/3 → nhìn app thật bấm chuyển đổi; AC4 → xác nhận nhãn đổi đúng theo chế độ; AC5 → grep không nhãn số mới; AC6 → `git diff` xác nhận không hàm `Log*` nào bị sửa.
- **Fixtures:** cần seed thủ công vài ngày dữ liệu giả trong CSDL test (không phải dữ liệu thật của chủ dự án) để verify AC1 nhanh, thay vì chờ 7 ngày thật.
- **Edge cases:** app mới cài (< 7 ngày lịch sử) → chỉ có N < 7 ngày dữ liệu, N ngày đầu tiên (chưa cài app) phải là gap chứ không phải "ngày 0 mẫu = ngày yên tĩnh giả"; ranh giới nửa đêm khi query "7 ngày gần nhất" lúc đang chạy gần 00:00.
