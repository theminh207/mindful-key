# 1.1: Harness Backspace Edge Cases

**Story ID:** 1.1
**Epic:** 1 — iOS Round 1 (Walking Skeleton)
**Slug:** harness-backspace-edge-cases
**Status:** done

## Story

As a **core-engine test author** (the engineer who maintains `tests/core/test_engine.cpp`),
I want **the harness to be able to inject a backspace (⌫) keystroke mid-sequence and read back
the resulting screen buffer**,
so that **backspace / mid-syllable-correction behavior is proven by a real regression test
before the iOS bridge work (story 1.2) starts touching `backspaceCount`**.

## Acceptance Criteria

1. `tests/core/test_engine.cpp` gains a way to express "press ⌫" inside a `runCase(...)` input
   string — a sentinel character is added to `KEYMAP` that maps to `KEY_DELETE`, and `typeChar`
   is extended to interpret the engine's response to that keystroke correctly (see Dev Notes —
   `vDoNothing` + delete must POP the screen buffer, not push a literal character).
2. At least **3 new regression cases from Loại 6 ("Backspace / sửa giữa từ")** of
   `.claude/skills/mindful-test-design/references/telex-vni-edge-cases.md` are added to `main()`
   as `runCase(...)` calls, each with an expected string that was transcribed from a real
   `make test-core` run (not guessed/remembered).
3. `make test-core` exits with code `0` and prints `gFail == 0` ("XONG — TẤT CẢ PASS") — i.e.
   all pre-existing cases (5 anchor cases + the 2026-07-11 dogfood lot) AND the new backspace
   cases pass together.
4. The exit-code gate itself (`return gFail == 0 ? 0 : 1;` in `main()`) is preserved and still
   functions — a deliberately-wrong expectation must still make `make test-core` fail with a
   non-zero exit code (verified locally, not committed).

   `tests/core/test_engine.cpp` (and `tests/core/build.sh` only if a new build flag turns out to

## Dev Notes

<!-- LOCKED. External dev tools must not edit Dev Notes. -->

- **Harness mechanics (what exists today):** `tests/core/test_engine.cpp` defines its own
  `KEYMAP` (ASCII char → `Uint16` keycode, only letters/digits/space/dot/comma today — no delete
  entry), a `typeChar(char c)` that calls `vKeyHandleEvent()` then interprets `st->code`
  (`vDoNothing` pushes the raw char; `vWillProcess`/`vRestore`/`vRestoreAndStartNewSession` pop
  `backspaceCount` chars then push `newCharCount` decoded chars), and `runCase(telex, expect)`
  which clears the virtual `screen`, calls `startNewSession()`, feeds every char of the input
  string through `typeChar`, and compares the resulting `screenStr()` to `expect`, incrementing
  `gFail` on mismatch. `main()` returns `gFail == 0 ? 0 : 1` so CI can gate on it — this exit-code
  gate was itself a 2026-07-11 fix (previously the harness always exited 0, i.e. "green" even with
  failing cases) [Source: .claude/skills/mindful-test-design/references/telex-vni-edge-cases.md
  lines 26-27]. [Source: tests/core/test_engine.cpp]
- **Why `typeChar` cannot just treat delete like any other key today:** `KEYMAP` has no entry
  mapping any ASCII char to `KEY_DELETE`, so there is currently no way to inject ⌫ into a
  `runCase` string at all. [Source: tests/core/test_engine.cpp lines 40-49]
- **What the engine actually returns for `KEY_DELETE` (read, not guessed):**
  `core/engine/Engine.cpp` handles `KEY_DELETE` explicitly (lines ~1440-1478): it sets
  `hCode = vDoNothing`, `hExt = 2` ("Delete key"), `hBPC = 0`, `hNCC = 0`, and mutates its
  *internal* buffer state (`_index--`, `_stateIndex--`, `startNewSession()` if `_index == 0`).
  This means the harness's `vDoNothing` branch — which today unconditionally does
  `screen.push_back((unsigned int)c)` — would currently push the sentinel delete character
  literally onto the virtual screen instead of removing the last character. The harness must
  special-case this itself; the engine will NOT report it via `backspaceCount`. [Source:
  core/engine/Engine.cpp lines 1440-1478]
- **Precedent to mirror (real iOS bridge already solved this exact problem):**
  `platforms/apple/ios/KeyboardExtension/KeyboardBridge.mm`'s `vDoNothing` case special-cases
  `keyCode == KEY_DELETE` and sets `result.backspaceCount = 1` manually — i.e. the *shell*
  (bridge/harness), not the engine's `hCode`, decides that a `vDoNothing` + delete-keycode means
  "remove one char". `typeChar()` in `test_engine.cpp` should follow the same interpretation:
  check the keycode being processed (not just `st->code`) and pop `screen` when it's
  `KEY_DELETE` and `st->code == vDoNothing`. [Source:
  platforms/apple/ios/KeyboardExtension/KeyboardBridge.mm lines 115-125]
- **`KEY_DELETE` value:** `22`, defined in `core/engine/platforms/linux.h` (the branch this
  Linux/macOS-built harness compiles against per `DataType.h`'s `#ifdef LINUX` / `#else mac.h`
  split — `tests/core/build.sh` passes `-DLINUX` only on Linux/WSL, none on macOS). [Source:
  core/engine/platforms/linux.h line 13]
- **Edge-case source material (Loại 6, "Backspace / sửa giữa từ ⭐ dễ vỡ nhất"):** the matrix
  lists 3 unverified (`🔬`) scenarios to cover: "Backspace giữa âm tiết: gõ `vieetj` rồi ⌫ 1 lần",
  "Sửa dấu sau khi lỡ: `vietj` → thêm `e` (chèn giữa buffer)", "⌫ về rỗng rồi gõ lại → không
  crash, `startNewSession` đúng". [Source:
  .claude/skills/mindful-test-design/references/telex-vni-edge-cases.md#Loại-6] These are
  explicitly marked "chưa chạy — verify riêng" — the matrix itself instructs: run
  `make test`, read the real output, only then lock the expectation; the matrix is "*bản đồ ca
  cần nghĩ tới*, KHÔNG phải danh sách kỳ vọng đã chốt" [Source:
  .claude/skills/mindful-test-design/references/telex-vni-edge-cases.md lines 7-11, 118-125].
- **FR this story supports:** FR-A17 (regression harness credibility) and, indirectly, FR-005 —
  `tests/ios`'s bridge-layer test is explicitly required to "reuse the exact Telex→Unicode case
  set of `tests/core/test_engine.cpp`, not invent new cases" [Source: tech-spec.md#FR-005], so
  keeping `test_engine.cpp` itself accurate and exit-code-honest before Mốc B/1.2 touches
  `backspaceCount` reduces the risk of the bridge silently diverging from engine truth. [Source:
  tech-spec.md#Testing-Strategy]
- **Hard constraint (from sharding context, applies to every 1.x story):** `core/` is FROZEN —
  `git diff core/` must be empty. Do NOT touch `platforms/apple/macos/*.mm` (macOS team's work in
  progress). This story's owned scope is limited to `tests/core/**`. [Source:
  bmad-output/ios/context/sharding-context.md#RÀNG-BUỘC-CỨNG]
- **Verify-don't-guess (project-wide discipline, doubly true here):** every new `expect` string
  in this story MUST come from reading real `make test-core` stdout, not from memory or from the
  matrix's `🔬` guesses — the matrix explicitly warns "Ca nào chưa chạy đánh dấu 🔬 (cần verify)"
  and "Engine nói sự thật — `make test` là trọng tài, không phải bảng này." [Source:
  .claude/skills/mindful-test-design/references/telex-vni-edge-cases.md lines 7-11, 124-125]
- **[Inference] Sentinel char choice:** the story does not mandate a specific ASCII character for
  the new `KEYMAP` delete entry. The literal ASCII backspace control character (`'\b'`, `0x08`)
  is a reasonable, self-documenting choice since it doesn't collide with any existing `KEYMAP`
  entry (letters, digits, space, `.`, `,`) — but this is the dev agent's implementation call, not
  a cited fact.
  Sequences that read confusingly as raw C string literals may be composed with string
  concatenation or explicit `\b` escapes in the `runCase(...)` call — again an implementation
  detail left to the dev agent, not prescribed here.
- **[Inference] Interpreting "chèn giữa buffer" (Loại 6, case 2):** the matrix's phrase implies
  the user goes back with ⌫ to *before* the point of error and inserts a corrective character,
  not merely appending one at the end of an already-finished word. Because `runCase` only exposes
  a linear left-to-right keystroke string, this must be expressed as an explicit ⌫-then-retype
  keystroke sequence (e.g. type a wrong string, ⌫ some number of times, then continue typing the
  correction) — the dev agent should design the exact sequence, run it, and record whatever the
  engine truthfully returns as the locked expectation, per the verify-don't-guess rule above.
- **Note for the orchestrator (citation/status caveat, not a blocker for authoring this story):**
  `bmad-output/ios/decision-log.md`'s 2026-07-11 entry ("Mốc B XONG… commit 91a8742") indicates
  the iOS `KeyboardBridge.mm` already implements Telex typing including `KEY_DELETE` handling in
  production code, ahead of this story's nominal "Blocks: 1.2" ordering. This story remains valid
  and unaddressed on its own terms — `tests/core/test_engine.cpp` today still has no way to
  inject ⌫ at all (verified by reading the file, see above) — but the sequencing implied by
  "Blocks: 1.2" may already be moot in practice. Flagging for the orchestrator's cross-story
  review rather than resolving unilaterally. [Source: bmad-output/ios/decision-log.md
  #2026-07-11 — "Mốc B XONG"]

## Testing

<!-- LOCKED. External dev tools must not edit Testing. -->

**Test type:** Hand-rolled harness, not a test framework — "tests" are ordered `runCase(...)`
calls inside `main()`, gated by the process exit code (`gFail == 0 ? 0 : 1`). No mocks/fixtures
needed: the harness is fully self-contained (no DB, no network, no filesystem I/O beyond the
compiled binary itself).

**Strategy:**
- Run `make test-core` after EVERY new `runCase` addition — never write an `expect` string before
  seeing the real `got` value printed by the harness. This is the project's standing
  "verify-don't-guess" rule and is doubly emphasized by the edge-case matrix itself.
- Confirm the pre-existing 13 locked cases (5 anchor + 8 dogfood-lot) still pass unchanged after
  the `typeChar`/`KEYMAP` edit — a delete-handling change that is scoped wrong (e.g. accidentally
  changes non-delete branches) would silently regress those.
- Scenarios that must be covered by the ≥3 new cases (AC #2), pulled from Loại 6 of the edge-case
  matrix:
  - Backspace mid-syllable after a completed vowel/tone transformation (e.g. building on
    `vieetj` → `việt`, then removing characters and confirming the buffer + underlying engine
    session stay consistent — no crash, no leftover phantom marks).
  - A correction sequence that inserts a character after backspacing partway through a word
    (not just trimming from the end) — this is the harness's first real test of "edit-in-place",
    which the current 13 locked cases never exercise.
  - Backspacing all the way to an empty buffer, then typing a fresh word — must not crash and
    must behave as a clean `startNewSession()` (the "⌫ về rỗng rồi gõ lại" case).
- Regression-gate self-check (AC #4): before considering the story done, deliberately break one
  `expect` string, confirm `make test-core` reports `❌ SAI`, a non-zero `gFail` count, and a
  non-zero process exit code — then revert. This proves the CI gate added on 2026-07-11 (which
  replaced an always-exit-0 harness) has not been silently weakened by this story's edits.
- Edge cases NOT in scope for this story (leave marked `🔬` in the matrix, do not attempt): Loại 5
  (invalid spelling chains), Loại 11 (VNI numeric marks), Loại 12 (stress/500-char strings) — none
  of these touch delete/backspace behavior and are out of this story's bounded scope.
- Final gate before marking done: `make test-core` exits 0, prints "XONG — TẤT CẢ PASS", and
  `git diff core/` is empty.
